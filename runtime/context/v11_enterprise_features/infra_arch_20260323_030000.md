## 1. Architectural Strategy for Sprint 11
This sprint transforms our CRUD application into a real-time, media-rich collaborative platform. The technical challenges involve handling binary data (files), deeply nested relationships (checklists), and maintaining state consistency across multiple active clients (WebSockets).

### 1.1 Real-Time Sync Strategy (WebSockets)
*   **Decision:** We will implement **Laravel Reverb** (introduced in Laravel 11) as our primary WebSocket server. It is natively integrated, performant (built on ReactPHP/Swoole concepts), and avoids the cost/vendor-lock-in of third-party services like Pusher, while remaining API compatible.
*   **Implementation:** 
    *   Backend: Broadcast `TaskUpdated`, `TaskMoved`, and `CommentAdded` events on private channels (e.g., `private-project.{project_id}`).
    *   Frontend: Use `Laravel Echo` with the `pusher-js` adapter configured to point to our local Reverb server.

### 1.2 File Storage Strategy
*   **Decision:** Use Laravel's standard filesystem abstraction. For MVP/Local dev, use the `local` or `public` disk. For GCP Production, we will configure the `gcs` (Google Cloud Storage) disk.
*   **Security:** Attachments must be stored securely. Direct public URLs should only be used if the bucket is secured, or URLs should be signed/served through a controller middleware to enforce RBAC.

## 2. Database Schema & API Contracts

### 2.1 Schema Updates (Migrations)

**1. Create `attachments` table (Polymorphic)**
To allow attaching files to Tasks (and potentially Comments or Projects later).
```php
Schema::create('attachments', function (Blueprint $table) {
    $table->id();
    $table->morphs('attachable'); // attachable_id, attachable_type
    $table->foreignId('user_id')->constrained()->cascadeOnDelete(); // Uploader
    $table->string('filename');
    $table->string('path'); // Storage path
    $table->string('mime_type');
    $table->unsignedBigInteger('size'); // Bytes
    $table->timestamps();
});
```

**2. Create `checklists` and `checklist_items` tables**
```php
Schema::create('checklists', function (Blueprint $table) {
    $table->id();
    $table->foreignId('task_id')->constrained()->cascadeOnDelete();
    $table->string('title')->default('Checklist'); // Allows multiple checklists per task later
    $table->timestamps();
});

Schema::create('checklist_items', function (Blueprint $table) {
    $table->id();
    $table->foreignId('checklist_id')->constrained()->cascadeOnDelete();
    $table->string('content');
    $table->boolean('is_completed')->default(false);
    $table->integer('position')->default(0); // For ordering
    $table->timestamps();
});
```

**3. Modify `tasks` table**
*   Update the `description` column (if not already `longText` or `text`) to ensure it can hold potentially large HTML/Markdown strings from the Rich Text Editor.

**4. Utilize core `notifications` table**
*   Run `php artisan notifications:table` to generate Laravel's default polymorphic notifications table.

### 2.2 API Endpoints

*   **Attachments:**
    *   `POST /tasks/{task}/attachments` (Multipart form data upload)
    *   `DELETE /attachments/{attachment}`
*   **Checklists:**
    *   `POST /tasks/{task}/checklists`
    *   `POST /checklists/{checklist}/items`
    *   `PATCH /checklist-items/{item}` (Toggle `is_completed`)
    *   `DELETE /checklist-items/{item}`
*   **Notifications:**
    *   `GET /notifications` (Fetch unread/recent)
    *   `PATCH /notifications/{notification}/read` (Mark as read)

## 3. Security & Infrastructure Handover
*   **DevOps / Infra:** 
    *   Need to ensure the GCP Cloud Run container allows WebSocket connections (HTTP/1.1 upgrade or HTTP/2).
    *   Configure a GCS bucket for `FILESYSTEM_DISK=gcs` and provide the necessary Service Account credentials to the environment.
    *   Run the Reverb server process (`php artisan reverb:start`) alongside the main web process, or deploy it as a separate Cloud Run service if high scale is required. (For MVP, `supervisor` running both in one container is acceptable).
