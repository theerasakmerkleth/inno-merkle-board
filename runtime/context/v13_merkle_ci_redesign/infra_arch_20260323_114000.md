# Infrastructure & Architecture Specification: Merkle CI (Enterprise Readiness)
**Project:** TaskFlow AI (Enterprise Task OS)
**Version:** 13.0 (Merkle CI Redesign)
**Date:** 2026-03-23
**Architect:** Enterprise Solution Architect (GCP & Laravel)

## 1. Requirements Analysis
Based on the **Sprint 11 PRD** and the **Merkle CI Design Spec**, the following architectural components are required to support "Enterprise Readiness":
-   **Real-time Collaboration:** Instant updates for Kanban card movement, comment notifications, and user presence.
-   **Rich Content Storage:** Secure storage and retrieval of file attachments and rich text descriptions.
-   **Scalability:** Support for high-density data and concurrent user interactions.

## 2. Data Modeling & Strategy

### 2.1 Database Schema (Confirmation)
The current schema already supports the core enterprise features:
-   `attachments`: Polymorphic table for tasks, comments, or projects.
-   `checklists` & `checklist_items`: One-to-many relationship with `tasks`.
-   `notifications`: Laravel's database-backed notification system.

**Optimization Needed:**
-   Ensure indexes on `attachments(attachable_id, attachable_type)` and `checklist_items(checklist_id)`.
-   Add an index to `tasks(board_id, status)` for faster Kanban rendering.

### 2.2 Rich Text Storage
-   **Strategy:** Descriptions and comments will be stored as sanitized HTML in the `text` columns.
-   **Sanitization:** Use `purifier` or similar to prevent XSS while allowing essential formatting tags (e.g., `<p>`, `<b>`, `<code>`).

## 3. Real-time Infrastructure (WebSockets)

### 3.1 Laravel Reverb Architecture
For the Merkle CI redesign, we will use **Laravel Reverb** to drive US-024 (Real-time Kanban Sync).
-   **Broker:** **GCP Memorystore (Redis)** will be used as the backplane for Reverb, ensuring state consistency across multiple Cloud Run instances.
-   **Protocol:** Secure WebSockets (WSS).
-   **Deployment:** Reverb will run as a sidecar container or a dedicated service on **GCP Cloud Run**, exposing port 8080 (standard for Cloud Run) and mapping to 6001 for WebSocket traffic.

## 4. File Storage Strategy (GCP)

### 4.1 Cloud Storage Integration
-   **Provider:** **Google Cloud Storage (GCS)**.
-   **Driver:** `spatie/laravel-google-cloud-storage`.
-   **Security:**
    -   Attachments will be stored in a private bucket (`merkle-task-assets`).
    -   Retrieval will use **Signed URLs** with a 15-minute expiration to ensure files are only accessible to authorized project members.
    -   Max file size limit: **10MB** (Enforced by Laravel `FormRequest` and GCS policy).

## 5. Terraform Infrastructure (main.tf)

The following GCP resources will be defined/updated:
-   `google_storage_bucket.attachments`: Private bucket for assets.
-   `google_redis_instance.cache_reverb`: High-availability Redis instance for session, cache, and Reverb broadcasting.
-   `google_cloud_run_v2_service.taskflow_api`: Configuration for Reverb environment variables (`REVERB_HOST`, `REVERB_PORT`, etc.).

## 6. API Contracts (Backend-Frontend Handoff)

### 6.1 WebSocket Events
-   **Channel:** `private-project.{id}`
-   **Event:** `TaskUpdated`
-   **Payload:** `{ task: TaskResource }` (Triggers on status, priority, or assignee change).

### 6.2 Attachment Endpoints
-   `POST /api/tasks/{task}/attachments` -> Upload file (Multipart).
-   `GET /api/attachments/{id}/download` -> Redirect to Signed URL.
-   `DELETE /api/attachments/{id}` -> Remove file and record.
