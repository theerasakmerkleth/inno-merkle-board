# UX/UI Design Specification: Agile Reporting & Dashboards
**Project:** TaskFlow AI (Enterprise Task OS)
**Version:** 27.0 (Agile Performance & Predictability)
**Date:** 2026-03-23
**Designer:** Senior UX/UI Architect & Lead Designer

## 1. Problem Statement & Strategic Vision
**Agile Context:** Delivering features is only half the battle. Enterprise leaders need to measure team velocity and predictable delivery timelines. Currently, TaskFlow AI lacks the analytical depth to show if a team is "on track" during a sprint or how their performance has trended over time.
- **Goal:** Design a high-performance "Reports" dashboard for every project that visualizes Burndown (execution health) and Velocity (historical capacity). Maintain the Merkle "Clean White" aesthetic with high-contrast data visualization.

## 2. Information Architecture (IA) Redesign

### 2.1 The Reports Hub (`/projects/{key}/reports`)
- **Integration:** A new "Reports" tab in the project header, alongside "Board", "Roadmap", and "Settings".
- **Layout:** A grid-based dashboard layout (`grid-cols-1 lg:grid-cols-2`) featuring large, interactive chart cards.

### 2.2 Core Reporting Components

#### A. Burndown Chart (The "Execution" Health)
- **Visuals:** A dual-line chart.
    - **Actual Remaining Work:** A solid **Cobalt Blue (`#0328D1`)** line showing the sum of tasks (or story points) not in "Done".
    - **Ideal Guideline:** A dashed **Merkle Red (`#DD3039`)** line connecting the start-of-sprint total to zero at the end-of-sprint date.
- **Interaction:** Hovering over any day on the X-axis shows a tooltip with "Remaining Tasks" and "Variance from Ideal".

#### B. Velocity Chart (The "Capacity" History)
- **Visuals:** A grouped bar chart comparing multiple boards (sprints).
    - **Commitment:** A light gray bar (`#AEAEBC`) showing work added to the board at start.
    - **Completed:** A vibrant **Sky Blue (`#0391F2`)** bar showing work moved to "Done" by the board end date.
- **X-Axis:** Lists the last 5-10 boards chronologically.
- **Y-Axis:** Total Task Count (or points).

## 3. UI Flow & State Management

### 3.1 Data Granularity
- Users can toggle between **"By Task Count"** and **"By Story Points"** (if implemented in future) via a small `<Tabs>` switcher at the top of the reports page.
- **Filter:** A date range picker to zoom into specific reporting periods.

### 3.2 Visual Language (Merkle CI)
- **Canvas:** Stark White (`#FFFFFF`).
- **Typography:** **Merkle Blue (`#040E4B`)** for all chart labels and axis titles to ensure AAA contrast.
- **Empty State:** If a project has no tasks with dates or multiple boards, show a "Not enough data to generate reports" illustration using Merkle CI grays.

## 4. Requirement Traceability
- **F25:** Burndown Charting (Current Board context).
- **F26:** Velocity Tracking (Cross-Board historical context).
- **F27:** PDF/PNG Export (Quick-action button in the header).

## 5. Developer Handoff
- **Library Recommendation:** `Recharts` (React) is highly recommended for its clean API and SVG rendering which aligns with our pixel-perfect needs.
- **API Requirements:** 
    - `GET /api/projects/{id}/stats/burndown`: Daily snapshots of task counts for the active board.
    - `GET /api/projects/{id}/stats/velocity`: Completed vs. Committed totals per board.
- **CSS:** Use `--primary` for Actual Burn and `--destructive` for the Ideal Guideline.
