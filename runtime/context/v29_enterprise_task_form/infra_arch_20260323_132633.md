# System Architecture & Technical Strategy: Enterprise Task Form Remediation

## Context
Based on the Code Review report `code_review_20260324.txt` and the PRD for the Jira-Style Enterprise Task Form (v29), the architecture requires adjustments to support missing features (Labels), enhance security (XSS sanitization), and improve frontend maintainability.

---

## 1. Database & Backend Architecture Updates

### 1.1 Data Modeling: Labels (Tags)
The DB migration `2026_03_23_055004_add_enterprise_fields_to_tasks_table.php` correctly added the `labels` column as a JSON field. The `Task` model has been updated to cast this to an `array`.

*   **Current State:** Backend supports `labels` as an array of strings.
*   **Action Required:** No database changes needed. The API endpoints (`TaskController@store`, `TaskController@update`) already validate `labels` as an array of strings. 

### 1.2 Security: HTML Sanitization (XSS Prevention)
The `description` field stores rich text HTML generated by TipTap. To prevent Cross-Site Scripting (XSS), the backend MUST sanitize this input before saving it to the database.

*   **Action Required (Backend Engineer):** 
    1.  Install the `mews/purifier` package via Composer (`composer require mews/purifier`).
    2.  Publish the configuration.
    3.  Update `TaskController@store` and `TaskController@update` to sanitize the `description` field using `clean($request->description)` before passing it to Eloquent.

### 1.3 Error Handling & API Responses
The Code Review flagged raw `fetch` calls without error handling in `handlePostComment`. While this is primarily a frontend issue, the backend must ensure it returns structured error responses (e.g., 422 for validation, 500 for server errors).

*   **Action Required (Backend Engineer):** Ensure API endpoints returning JSON (like the comment endpoints) strictly adhere to Laravel's standard error response format.

---

## 2. Frontend Architecture & Component Strategy

The `KanbanBoard.tsx` and specifically the `TaskModal` have grown too complex (High Cyclomatic Complexity). We must refactor this into a modular architecture to adhere to clean code principles.

### 2.1 Component Extraction Map (React)
The Frontend Engineer MUST refactor `TaskModal` into the following atomic components within `resources/js/Components/Task/`:

1.  `TaskModal.tsx` (Main Container)
    *   Manages the `<Sheet>` layout, overall `useForm` state, and save/delete actions.
2.  `TaskMetadataSidebar.tsx`
    *   **Props:** `data`, `setData`, `columns`, `project_members`, `canEdit`, `errors`.
    *   **Responsibility:** Renders the right-hand sidebar containing Status, Assignee, Priority, Story Points, Dates, and the newly required **Labels** multi-select.
3.  `TaskDescriptionEditor.tsx`
    *   **Props:** `content`, `onChange`, `canEdit`.
    *   **Responsibility:** Encapsulates the TipTap editor logic and the custom toolbar.
4.  `TaskChecklist.tsx`
    *   **Props:** `taskId`, `checklists`, `canEdit`.
    *   **Responsibility:** Handles adding, toggling, and deleting sub-tasks.
5.  `TaskComments.tsx`
    *   **Props:** `taskId`, `comments`, `auth_user`.
    *   **Responsibility:** Renders the comment timeline and the input area. MUST include proper error handling (try/catch, toast notifications) for the API call.

### 2.2 Implementing the Missing "Labels" Feature
*   **Action Required (Frontend Engineer):** 
    *   Integrate a multi-select component (e.g., `react-select` which is already installed) into `TaskMetadataSidebar.tsx`.
    *   Bind it to `data.labels` in the Inertia form state.
    *   Ensure it visually matches the Merkle CI badge style (`bg-accent text-accent-foreground`).

### 2.3 Performance Optimization
*   **Action Required (Frontend Engineer):** Wrap `TaskCard` in `React.memo` to prevent unnecessary re-renders during drag operations across large boards.

## 3. API Contracts (Reminders)

### POST /tasks/{task}/comments
*   **Request:** JSON `{ "content": "string" }`
*   **Response (Success):** `201 Created` with the comment object.
*   **Frontend Handling:** Must check `res.ok`. If false, display an error toast and do not clear the input field.

### PATCH /tasks/{task}
*   **Request (Labels):** `{ "labels": ["backend", "urgent"] }`
*   **Backend Validation:** `'labels' => 'nullable|array', 'labels.*' => 'string|max:50'`
