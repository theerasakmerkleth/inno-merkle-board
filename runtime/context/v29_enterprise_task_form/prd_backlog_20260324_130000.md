# Product Requirements Document (PRD): Enterprise-Grade Task Management Form (Jira-Style)

## 1. Value Proposition
**Goal:** To elevate the existing task creation and editing experience to an "Enterprise-Grade" level, mirroring the detail, flexibility, and power of industry standards like Jira.
**ROI:** Increases data fidelity, improves cross-functional communication by allowing rich context (TipTap full-text editor, attachments, detailed metadata), and streamlines workflow management for complex projects by capturing all necessary tracking dimensions (e.g., Story Points, Labels, Linked Issues) directly at the point of creation or edit.

## 2. User Empathy & Journeys
**Persona:** Team Members (Developers, QA, PMs)
**Journey:** A PM needs to log a detailed bug or a comprehensive user story. They click "Create Card" and are presented with a robust, well-organized form. Instead of just a basic description, they can use a full-text editor to format text, add tables, or embed code snippets. They can categorize the task with labels, set priority, estimate story points, and attach screenshots directly within the same fluid modal interface.

## 3. Epics & User Stories

### Epic 1: Jira-Style Enterprise Task Form
The current task modal needs to be expanded to handle comprehensive metadata and rich content without overwhelming the user interface.

#### US-1.1: Comprehensive Task Metadata Fields
**Story:** *As a Team Member, I want to input detailed metadata for a task, so that all project context is captured centrally.*
* **AC 1:** The Create/Edit form MUST include the following fields: 
  * `Title` (Required, String)
  * `Description` (Rich Text Editor, see US-1.2)
  * `Status` / `Column` (Dropdown)
  * `Assignee` (Dropdown of project members)
  * `Reporter` (Read-only in edit mode, auto-set to creator)
  * `Priority` (Dropdown: Lowest, Low, Medium, High, Highest)
  * `Story Points` / `Estimation` (Number input)
  * `Start Date` & `Due Date` (Date pickers)
  * `Labels` / `Tags` (Multi-select component or text input with pills)
* **AC 2:** The form layout should be organized logically, potentially splitting into a main content area (Title, Description, Attachments) and a right-hand sidebar for metadata (Status, Assignee, Dates, Points), mimicking enterprise tools.

#### US-1.2: Advanced Full-Text Editor (TipTap Integration)
**Story:** *As a Team Member, I want a powerful rich-text editor for the description, so that I can format requirements clearly.*
* **AC 1:** The Description field MUST utilize an advanced TipTap editor instance.
* **AC 2:** The editor MUST support: Bold, Italic, Strikethrough, Headings (H1, H2, H3), Bulleted Lists, Numbered Lists, Blockquotes, Code Blocks, and Hyperlinks.
* **AC 3:** The editor toolbar must be persistently visible while the field is focused or always visible.
* **AC 4:** The content must be safely sanitized before rendering to prevent XSS.

#### US-1.3: Attachment Support in Form
**Story:** *As a Team Member, I want to attach files directly from the creation/edit form, so that visual context is immediately available.*
* **AC 1:** A Drag-and-Drop area MUST be available for attachments within the form.
* **AC 2:** Users can drop files (images, PDFs, text files) or click to browse.
* **AC 3:** Uploaded files appear as thumbnails or file links below the drop zone.
* **AC 4:** Files can be deleted from the task during the edit mode.

#### US-1.4: Optimistic UI & Seamless Saving
**Story:** *As a Team Member, I want the form to save changes efficiently without jarring page reloads, so that my workflow remains uninterrupted.*
* **AC 1:** When clicking "Save" or "Create", the form submission uses Inertia's `preserveScroll` and `preserveState` to maintain the user's position on the Kanban board.
* **AC 2:** Validation errors (e.g., missing Title) are displayed inline immediately beneath the respective fields.
* **AC 3:** If in edit mode, modifying a field (like Status) and saving updates the underlying Kanban board state seamlessly.

## 4. Edge Cases & Failure Modes
* **Validation Failure:** If a user submits a task without a title, the form remains open, the title field turns red, and an explicit error message is shown.
* **Large File Uploads:** If an attachment exceeds the server's MAX_FILE_SIZE limit, a graceful error toast is presented to the user, and the file is rejected before a full upload completes.
* **Permission Constraints:** If a user with a "Viewer" role attempts to open the form, it renders in a strictly read-only state.

## 5. Non-Functional Requirements (NFRs)
* **Accessibility:** All form inputs must have associated `<label>` tags (using `htmlFor`). The modal must trap focus and be dismissible via the `Escape` key.
* **Responsiveness:** The enterprise form, despite having more fields, must stack gracefully on mobile viewports (e.g., sidebar metadata moves below the main description).
