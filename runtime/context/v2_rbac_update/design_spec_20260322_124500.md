## 1. Service Blueprint & User Journey

### 1.1 Global Administration Flow
- **User (Admin):** Logs in -> Navigates to `Settings > User Management`.
- **Front-stage (React/shadcn):** 
    - Displays a `<DataTable>` listing all workspace users.
    - Top bar features an `Invite User` `<Button>`.
    - Each row contains a `<Select>` dropdown for assigning the Global Role (Admin, PM, Developer, QA, Agent) and a `<Switch>` for instant access toggling.
- **Back-stage (Laravel API):** `GET /api/v1/users`, `POST /api/v1/users/invite`, `PATCH /api/v1/users/{id}/role`.
- **Support System (Database):** Spatie Permission tables (roles, model_has_roles).

### 1.2 Project Setup Flow
- **User (PM):** Creates Project -> Navigates to `Project > Team`.
- **Front-stage (React/shadcn):**
    - Uses a dual-pane layout or `<Dialog>` modal to search workspace users and add them to the project.
    - A `<Select>` dropdown is used to assign Project-Level Roles (e.g., Viewer, Contributor, Manager).
- **Back-stage:** `POST /api/v1/projects/{pid}/members`.
- **Support System:** Pivot table mapping users to projects with roles.

### 1.3 Kanban Execution Flow (Jira-style RBAC)
- **User (Developer/QA):** Views the `<KanbanBoard>`.
- **Front-stage (React/dnd-kit/shadcn):**
    - The drag-and-drop (`dnd-kit`) engine must check the user's project role.
    - If a **Developer** drags a card from `In Progress` over `Done`, the drop zone is visually disabled (e.g., using `opacity-50 cursor-not-allowed` tailwind classes) and the drop event is cancelled.
    - Only **PM** or **QA** can successfully drop into `Done`.
- **Back-stage:** `PATCH /tasks/{id}/status` (Must validate user permissions before executing).

## 2. UI/UX Wireframes & Component Mapping

### 2.1 User Management Screen (`/users`)
*   **Layout:** Standard Dashboard layout with Sidebar.
*   **Header:** Page title "Team Members" with Material Icon `group`.
*   **Table (`shadcn/DataTable`):**
    *   Columns: Name/Avatar, Email, Global Role, Status (Active/Disabled), Actions.
    *   Role Selector (`shadcn/Select`): Dropdown with values: Admin, PM, Dev, QA, Agent.
    *   Status Toggle (`shadcn/Switch`): For immediate access revocation.
*   **Invite Modal (`shadcn/Dialog`):** Contains `shadcn/Input` for email and `shadcn/Select` for initial role.

### 2.2 Updated Kanban Board (`/projects/{id}/board`)
*   **Card Styling (`shadcn/Card`):** 
    *   Added visual cues for permissions. If a column is restricted (e.g., `Done` column for a Developer), the column header displays a tiny lock icon (`lock` Material Icon).
    *   When dragging, restricted columns highlight in light red (`bg-red-950/20 border-red-900` in dark mode) to indicate an invalid drop target.

## 3. Visual & Theming Guidelines (Merkle CI)
*   **Palette:** Background `bg-slate-950`, Surfaces `bg-slate-900`, Text `text-zinc-100`.
*   **Accents:** Primary Actions and Warnings use Deep Red (`bg-red-700 hover:bg-red-800`).
*   **Icons:** Material Icons.
    *   `shield` for Admin/Security settings.
    *   `group` for User Management.
    *   `lock` for restricted Kanban columns.
    *   `smart_toy` for AI Agents.

## 4. Handover to Architect
The UI now heavily relies on the backend correctly identifying both **Global Roles** and **Project-Level Roles**.
1.  **Requirement:** Please design a robust RBAC database schema. We recommend leveraging the concepts from packages like `spatie/laravel-permission` combined with project-specific overrides.
2.  **API Need:** The API must return the current user's permissions alongside the Kanban data so the React UI knows which columns to lock/unlock for drag-and-drop.
