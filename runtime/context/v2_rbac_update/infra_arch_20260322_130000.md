## 2. Database Schema Design (RBAC Update)

To support both Global and Project-Level permissions (Jira-style), we will implement a dual-layer RBAC system. 

### 2.1 Core Entities & Global RBAC (Using spatie/laravel-permission structure)

**`users` Table**
- `id` (PK)
- `name` (String)
- `email` (String, Unique)
- `is_active` (Boolean, Default: true) - Used by Admin to instantly disable access.
- `git_username` (String, Nullable)
- `timestamps`

**`roles` & `permissions` Tables (Global)**
- Standard Spatie tables (`roles`, `permissions`, `model_has_roles`, `role_has_permissions`).
- **Global Roles:** `Admin`, `Project Manager`, `Developer`, `QA`, `AI Agent`.
- **Global Permissions:** `manage_users`, `create_projects`, `view_all_projects`.

### 2.2 Project-Level RBAC

**`projects` Table**
- `id` (PK)
- `name` (String)
- `status` (Enum: 'active', 'archived')
- `timestamps`

**`project_user` Table (Pivot)**
- `id` (PK)
- `project_id` (FK -> projects.id)
- `user_id` (FK -> users.id)
- `project_role` (Enum: 'Manager', 'Contributor', 'Viewer') - Defines what they can do *within* this specific project.
- `timestamps`

### 2.3 Task Entity Update

**`tasks` Table**
- `id` (PK)
- `project_id` (FK -> projects.id)
- `title` (String)
- `description` (Text, Nullable)
- `status` (Enum: 'todo', 'in_progress', 'awaiting_review', 'qa_ready', 'done') *(Added `qa_ready`)*
- `assignee_id` (FK -> users.id, Nullable)
- `timestamps`

## 3. Security & Permission Logic (API Contracts)

### 3.1 Authorization Matrix (Backend Enforcement)
The Laravel backend (FormRequests/Policies) MUST enforce the following rules before processing any state change (`PATCH /tasks/{id}/status`):
- **Developer (`Contributor` role):** Can transition tasks to `in_progress` and `awaiting_review` / `qa_ready`.
- **AI Agent (`Contributor` role):** Can transition tasks to `awaiting_review` (via API).
- **QA (`Contributor` or `Manager` role):** Can transition tasks to `done`.
- **PM (`Manager` role):** Can transition tasks to any state and assign members to the project.
- **Admin:** Has global override access.

### 3.2 UI Permission Hydration
To support the Designer's requirement for disabling Kanban columns, the Kanban load endpoint (`GET /projects/{id}`) must include a `permissions` object for the current user:

```json
// Example payload injected via Inertia
{
  "project": { "id": 1, "name": "TaskFlow Core" },
  "columns": [ ... ],
  "user_permissions": {
    "can_move_to_done": false,
    "can_manage_team": false
  }
}
```
