# Product Requirements Document (PRD): Fix PDF Export in Agile Insights

## 1. Value Proposition
**Goal:** To resolve a critical bug preventing the "Export Dashboard to PDF" functionality from working on the Agile Insights (Reports) page.
**ROI:** Restores the core value proposition of Epic 2 (Enterprise Export Capabilities) delivered in v35. Stakeholders must be able to reliably generate offline PDF snapshots of project telemetry. A broken export feature degrades trust in the platform's enterprise readiness.

## 2. User Empathy & Journeys
**Persona:** Project Managers & Stakeholders (C-Level).
**Journey (Failure Mode):** The Project Manager navigates to the Reports page, clicks the "..." menu, and selects "Export Dashboard (PDF)". Instead of a PDF downloading, the toast notification either hangs on "Generating PDF..." or throws a "Failed to generate PDF" error. This blocks them from sharing the report.

## 3. Epics & User Stories

### Epic 1: Remediation of Client-Side PDF Generation

#### US-1.1: Reliable PDF Capture of Dashboard Container
**Story:** *As a Project Manager, I want the "Export Dashboard (PDF)" button to consistently capture the charts and tables and download a valid `.pdf` file, so that I can share the insights offline.*

* **AC 1 (Capture Target):** Given the user clicks "Export Dashboard (PDF)", When the client-side script executes, Then it MUST accurately target the DOM node containing the charts and tables (excluding the global header/navigation if possible, or handling the full page scroll height correctly).
* **AC 2 (Canvas Rendering):** Given the dashboard contains SVG elements (Recharts), When `html2canvas` (or equivalent) renders the view, Then the charts MUST be visible in the final image, not blank or corrupted.
* **AC 3 (PDF Generation):** Given the canvas is successfully captured, When `jsPDF` processes the image, Then the image MUST be scaled correctly to fit an A4 landscape page without being cropped or distorted.
* **AC 4 (Error Handling):** Given an unexpected rendering error occurs (e.g., a cross-origin image blocks canvas tainting), When the export fails, Then the UI MUST catch the error, remove the "Generating..." toast, and display an actionable "Failed" toast, logging the error to the console for debugging.

## 4. Edge Cases & Failure Modes
* **Viewport Scrolling:** If the dashboard content extends beyond the current visible viewport height (e.g., the user has scrolled down), `html2canvas` might only capture the visible area or render the scrolled area as blank black/white space. The frontend engineer must ensure the *entire* target container is captured regardless of scroll position.
* **SVG Rendering Issues:** Some charting libraries (like Recharts) use complex SVG `<clipPath>` or CSS transforms that `html2canvas` struggles to parse natively.
* **Resolution/Blurriness:** If the captured canvas is low resolution, the resulting PDF will look unprofessional. Ensure `scale: 2` (or higher) is applied for retina displays.

## 5. Non-Functional Requirements (NFRs)
* **Performance:** The browser main thread should not freeze for more than 5 seconds during PDF generation.
* **Dependencies:** If `html2canvas` proves too unreliable for SVG charts, the Engineering team is authorized to evaluate alternatives like `dom-to-image` or a backend headless browser solution (e.g., Puppeteer/Browsershot) as a fallback strategy.
