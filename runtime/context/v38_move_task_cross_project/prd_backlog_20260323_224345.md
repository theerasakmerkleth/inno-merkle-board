# Product Requirements Document (PRD): Cross-Project/Board Task Movement

## 1. Value Proposition
**Goal:** Allow users to move a task (card) from its current board to a different board, either within the same project or across different projects.
**ROI:** Increases flexibility for agile teams. Sometimes tasks are logged in the wrong project or need to be handed off to another team's board. Without this feature, users must manually recreate the task and delete the old one, losing comments and history.

## 2. User Empathy & Journeys
**Persona:** Project Managers, Developers.
**Journey:** A developer realizes a task in the "Frontend" project actually belongs to the "Backend" project. They open the Task Modal, click "Move", select the "Backend" project, select the "Sprint 2" board, and select the "To Do" column. The task disappears from the Frontend board and is transferred with all its metadata to the Backend board.

## 3. Epics & User Stories

### Epic 1: Cross-Project Task Handoff

#### US-1.1: Move Task API and Logic
**Story:** *As a Developer, I want the backend to support updating a task's project, board, and column, so that data integrity is maintained during a move.*
* **AC 1:** `TaskController@update` (or a dedicated `move` endpoint) MUST accept `project_id`, `board_id`, and `board_column_id`.
* **AC 2:** The backend MUST verify the user has access to the destination `project_id` and `board_id`.
* **AC 3:** If the task is moved, it MUST append to the bottom of the destination column.

#### US-1.2: Dynamic Destination Fetching
**Story:** *As a Frontend UI, I need to fetch the available boards and columns for a selected project, so the user can choose exactly where the task goes.*
* **AC 1:** An API endpoint `GET /api/projects/{project}/structure` MUST return the boards and their respective columns for the given project.

#### US-1.3: Move Task UI
**Story:** *As a User, I want a "Move" button in the Task Modal that opens a simple dialog to select the destination.*
* **AC 1:** The `TaskModal` footer MUST include a "Move" button (or it can be an action in the header).
* **AC 2:** Clicking "Move" opens a nested `<Dialog>` (or replaces the modal content) with dropdowns for Project, Board, and Column.
* **AC 3:** Submitting the move action triggers the backend update. On success, the task is removed from the current Inertia board state (or the page is refreshed).

## 4. Edge Cases & Failure Modes
* **Lost Assignees:** If the task is moved to a project where the current assignee is not a member, the assignee might need to be cleared, or the system should warn the user. For MVP, simply clearing the `assignee_id` if they aren't in the new project is acceptable, or keeping it but showing them as "External". We will keep the assignee for now but it's an accepted quirk if they lack permissions.
* **Permission Denied:** User tries to move to a project they can see but can't edit. The backend must reject with 403.

## 5. Non-Functional Requirements (NFRs)
* **Security:** The backend must strictly validate ownership of the destination entities.
