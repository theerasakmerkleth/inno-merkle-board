# UI/UX Design Specification: Cross-Project & Cross-Board Movement Refinement

## 1. UX Architecture & Brand Alignment

**Design Philosophy:** Moving a task across boards or entirely different projects is a high-stakes, macro-level action. It should not be buried alongside standard metadata like "Priority" or "Story Points". Currently, the "Location" dropdowns (Project, Board) sit directly in the right-hand metadata sidebar. This clutters the UI for daily use (since tasks are rarely moved cross-project) and violates the principle of Progressive Disclosure. 

**Key Objectives:**
*   **Progressive Disclosure:** Remove the "Location" (Project/Board) dropdowns from the default view of the right-hand `TaskMetadataSidebar`. 
*   **Explicit Intent:** Move this functionality into a dedicated "Move" or "Transfer" action to ensure the user clearly understands they are changing the task's parent context.
*   **Visual Separation:** We will introduce a dedicated `MoveTaskDialog` or a distinct "Move" action within the `...` (More Actions) menu or header of the `TaskModal`.

## 2. Core Views UI Specification

### 2.1 The Task Modal Header (`TaskModal.tsx`)
*   **Current State:** Contains the Task Key (e.g., `CORE-123`), Title input, and Close button.
*   **Redesign:** Add an "Actions" menu (`...` icon or a subtle "Move" button) to the header, specifically for overarching actions like "Move" or "Delete".

### 2.2 The "Move Task" Flow
*   **Trigger:** User clicks a "Move" button (Ghost style, `material-icons: shortcut`) in the header or next to the breadcrumb/task ID.
*   **Interaction:** Opens a `<Dialog>` or `<Popover>` strictly dedicated to the transfer process.
*   **UI Inside Dialog/Popover:**
    *   **Title:** "Move Task"
    *   **Description:** "Select a new project and board destination."
    *   **Field 1: Project Select:** Searchable dropdown (`react-select` or standard `<select>`).
    *   **Field 2: Board Select:** Dynamically populated based on Project selection.
    *   **Field 3: Column Select (Status):** Dynamically populated based on Board selection.
    *   **Actions:** "Cancel", "Confirm Move" (Primary Button).

### 2.3 The Metadata Sidebar (`TaskMetadataSidebar.tsx`) Cleanup
*   **Action:** Completely REMOVE the "Location" section (Project and Board dropdowns) from the right-hand sidebar.
*   **Reasoning:** The sidebar should only contain properties that modify the *state* of the task within its current board (Status, Assignee, Priority, Points, Dates, Labels). 

## 3. Interaction States & Edge Cases

*   **Loading State (Cross-Project Fetching):** When the user selects a new Project in the Move Dialog, a skeleton loader or a `spinner` must appear over the Board/Column selects while the `GET /api/projects/{key}/structure` request resolves.
*   **Permission Checks:** If the user selects a project where they only have "Viewer" access, the "Confirm Move" button must be disabled, or the project shouldn't appear in the list at all.
*   **Optimistic UI:** When "Confirm Move" is clicked, immediately close the modal and trigger a `toast.success("Task moved successfully")`. If the user is currently viewing the *source* board, the task should vanish. If the API fails, it should reappear with a `toast.error`.
