## 2. Database Schema Design (Telemetry & Auth Update)

### 2.1 Core Entities & Global RBAC 
*   **`users` Table:** Remains the same (contains `is_active`, `git_username`).
*   **Authentication:** We will utilize Laravel's native session-based authentication for the web UI (Inertia). AI Agents will continue to use Sanctum tokens.

### 2.2 Project-Level RBAC & Tasks
*   **`projects` Table:** Remains the same.
*   **`project_user` Table:** Remains the same.
*   **`tasks` Table:** Remains the same.

### 2.3 Telemetry Entity (NEW)

To fulfill the PMO directive for KPI Tracking (Cycle Time), we introduce the `task_transitions` table.

**`task_transitions` Table**
- `id` (PK)
- `task_id` (FK -> tasks.id)
- `user_id` (FK -> users.id, Nullable) - The user or agent who made the change.
- `from_status` (String, Nullable) - The previous status (null if it's task creation).
- `to_status` (String) - The new status.
- `created_at` (Timestamp) - Exact time of the transition.

## 3. Security & Permission Logic (API Contracts)

### 3.1 Authentication Middleware (NEW)
- All web routes (`/`, `/users`, `/projects/*`) MUST be wrapped in the `auth` middleware.
- Unauthenticated users must be redirected to `/login`.

### 3.2 Authorization Matrix (Backend Enforcement)
*(Rules remain the same as v3.0, but now enforced against the authenticated `auth()->user()` instead of dummy data).*

### 3.3 Telemetry Event Trigger (Backend Implementation)
Instead of manually inserting records in every controller, implement an Eloquent Model Event or Observer on the `Task` model.
- **Hook:** `updated`
- **Logic:** `if ($task->wasChanged('status')) { insert into task_transitions... }`
