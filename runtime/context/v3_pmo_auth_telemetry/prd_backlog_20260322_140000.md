## 1. Product Vision & Strategy
TaskFlow AI is designed to be the definitive enterprise task management platform, bridging the gap between human creativity and AI efficiency. Inspired by the **Merkle** design philosophy, the platform provides a seamless "Human-in-the-loop" experience backed by **Enterprise-grade Role-Based Access Control (RBAC)** mirroring Jira's robust permission schemes.

**Target Audience:** Enterprise Software Teams, DevOps Engineers, System Administrators, and Autonomous AI Workers.

**Pain Points Addressed:**
- Lack of granular access control and data security across multiple teams.
- Manual tracking of code deployments and task statuses.
- Inefficient collaboration and handovers between AI Agents and Human Reviewers.
- **[NEW]** Lack of portfolio-level visibility and KPI telemetry (Cycle Time).

## 2. User Stories & Acceptance Criteria (AC)

### 2.1 System Administrator - Global Security & Auth
**Story (US-001):** *As a System Administrator, I want to define global roles and permissions, so that I can control who has access to specific system features and projects.*
*   **AC 1:** System must support at least 5 default global roles: `Admin`, `Project Manager`, `Developer`, `QA`, and `AI Agent`.
*   **AC 2:** Admin can invite new users to the workspace and assign them a default global role.
*   **AC 3:** Admin dashboard for user management must allow enabling/disabling user access instantly.

**Story (US-002) [NEW/CRITICAL]:** *As a Workspace User, I want a secure login portal so that the system accurately recognizes my role and permissions across the app.*
*   **AC 1:** Users must log in via a standard email/password form (Laravel standard auth).
*   **AC 2:** Unauthorized users attempting to access `/` or `/users` must be redirected to the login page.
*   **AC 3:** The UI must reflect the permissions of the *currently logged-in user*, replacing all hardcoded dummy data.

### 2.2 Project Manager (PM) - Project-Level Control & Strategic Oversight
**Story (US-003):** *As a PM, I want to manage my project's team members and monitor resource loading, so I can ensure secure and efficient delivery.*
*   **AC 1:** PM can assign workspace users to specific projects with Project-Level Roles (e.g., a Developer in Project A might be a Viewer in Project B).
*   **AC 2:** Dashboard must display a "Resource Loading" heat map showing task distribution among project members.
*   **AC 3:** Only users explicitly added to a project can view or interact with its Kanban board.

### 2.3 Developer / QA - Targeted Workflow & Telemetry
**Story (US-004):** *As a Developer/QA, I want to access only the projects and tasks assigned to me, so I can focus on my work without noise or security risks.*
*   **AC 1:** Users can only transition task statuses (e.g., to "In Progress" or "QA Ready") if their assigned role permits it.
*   **AC 2:** Task status must move automatically via Git Webhooks (Developer commits) or API (QA automation scripts).
*   **AC 3:** Developers cannot move tasks to "Done" directly; tasks must pass through "Awaiting Review" or "QA Ready" first.

**Story (US-005) [NEW]:** *As a PMO Analyst, I want the system to automatically log every state transition of a task, so that I can measure Cycle Time KPIs.*
*   **AC 1:** Every drag-and-drop or API status change must insert a record into a `task_transitions` table.
*   **AC 2:** The record must capture the task ID, old status, new status, user ID who made the change, and the exact timestamp.

### 2.4 AI Agent - Seamless Integration
**Story (US-006):** *As an AI Agent, I want a robust RESTful API to pull assigned tasks and submit results, so I can function as a productive team member.*
*   **AC 1:** API must support OAuth2/Sanctum token authentication linked to an `AI Agent` role.
*   **AC 2:** `GET /api/v1/agent/tasks` returns only tasks assigned to the specific Agent ID.
*   **AC 3:** Submissions via API must trigger a "Human-in-the-loop" review notification. Agents cannot bypass the `awaiting_review` state.

## 4. Work Flow Revision (Jira-style)
1.  **Admin** provisions Workspace and invites **PMs / Devs / QAs**.
2.  **PM** creates a Project and adds team members (assigning Project-Level roles).
3.  **PM** creates Tasks and assigns them to **Devs** or **AI Agents**.
4.  **Dev/Agent** works on the task and transitions it to `Awaiting Review` or `QA Ready`. (Role restricted action). *System logs transition.*
5.  **QA/PM** reviews and transitions the task to `Done`. *System logs transition.*
