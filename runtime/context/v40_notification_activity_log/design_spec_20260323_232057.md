# UI/UX Design Specification: Notifications & Activity Log (Enterprise Best Practices)

## 1. UX Architecture & Brand Alignment

**Design Philosophy:** In a high-density, collaborative environment like MerkleBoard, users need a clear, non-intrusive way to track what changed, who changed it, and what requires their immediate attention. We must strictly separate **Notifications** (actionable items demanding the user's attention, like mentions or assignments) from the **Activity Log** (a passive, chronological audit trail of all changes).

**Key Objectives:**
*   **Separation of Concerns:** 
    *   **Notifications:** Global, urgent, accessible from the Top App Bar.
    *   **Activity Log:** Contextual, passive, attached to specific entities (e.g., inside the Task Modal).
*   **Cognitive Ease:** Use iconography and color (Merkle Cobalt/Red) sparingly to highlight actionable items. Use muted typography for historical audit data.
*   **Progressive Disclosure:** Do not overwhelm the user. Show recent activity first, with the ability to "Load More".

## 2. Core Views UI Specification

### 2.1 Global Notifications (The "Inbox")
*   **Trigger:** The existing "Bell" icon in the Top App Bar (`AppLayout.tsx`).
*   **Component:** `shadcn/ui` `<Popover>` or `<Sheet>` sliding from the right. (A Sheet is preferred for Enterprise apps to allow deeper reading without closing on accidental outside clicks).
*   **UI Layout (The Sheet):**
    *   **Header:** Title "Notifications" with a "Mark all as read" ghost button.
    *   **Tabs:** "All", "Unread", "Mentions".
    *   **Notification Item (Row):**
        *   **Avatar:** The user who triggered the event.
        *   **Content:** Rich text formatting. E.g., "**John Doe** assigned you to **[CORE-123] Secure Login API**".
        *   **Timestamp:** Relative time (e.g., "2 hours ago").
        *   **Unread Indicator:** A solid Merkle Red (`#E3000F` / `bg-destructive`) dot (8px x 8px) on the left edge.
        *   **Interaction:** Clicking the row marks it as read and opens the corresponding Task Modal or Project.

### 2.2 Contextual Activity Log (Inside Task Modal)
*   **Location:** Integrated into the `TaskModal.tsx`.
*   **Placement Strategy:** The Activity Log should share space with the Comments section, as both represent the chronological history of the task. 
*   **Component:** A Tabbed interface at the bottom of the "Main Content Area" (below the Description/Attachments).
    *   **Tabs:** `<Tabs defaultValue="comments"> <TabsList><TabsTrigger value="comments">Comments</TabsTrigger><TabsTrigger value="history">History</TabsTrigger></TabsList>...`
*   **UI Layout (History Tab):**
    *   **Format:** A vertical timeline (stepper) without heavy borders.
    *   **Activity Item (Row):**
        *   **Icon:** A small, muted icon indicating the action type (e.g., `edit`, `person_add`, `check_circle`).
        *   **Content:** Semantic sentence. E.g., "Changed status from **To Do** to **In Progress**".
        *   **Metadata:** By **Jane Smith** • 12 Mar, 14:30.
        *   **Styling:** Text is `text-xs text-muted-foreground`. Highlights (old/new values) are `font-medium text-foreground`.

## 3. Interaction States & Edge Cases

*   **Empty States:**
    *   *Notifications:* An illustration of an empty inbox with text: "You're all caught up! No new notifications."
    *   *Activity Log:* "No activity recorded yet."
*   **Real-time Updates (Optimistic UI):**
    *   If Laravel Reverb (WebSockets) is active, the Bell icon should immediately increment its red badge counter when a new notification arrives.
*   **Pagination:** 
    *   The Activity Log should initially load the last 10-20 events. A subtle "Load older activity..." button appears at the bottom if more records exist.
