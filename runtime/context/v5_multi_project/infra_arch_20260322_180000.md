## 1. Architectural Strategy

The shift from a single global workspace to a Jira-like Multi-Project/Multi-Board architecture requires a fundamental restructuring of our relational database model. 

The core challenge is implementing the **Project Key** system (e.g., `CORE-1`, `CORE-2`) where task IDs increment sequentially *per project*, rather than relying solely on the global auto-incrementing primary key.

---

## 2. Database Schema Redesign

### 2.1 The `projects` Table Update
We must add a unique `key` identifier used as the prefix for all tasks.
*   `id` (PK)
*   `key` (String, Unique, length: 3-5 chars, e.g., 'CORE', 'MKT') **[NEW]**
*   `name` (String)
*   `task_sequence` (Integer, Default: 0) **[NEW]** - Acts as a counter for generating the next task number (`CORE-1`, `CORE-2`).
*   `timestamps`

### 2.2 The `boards` Table (NEW)
Projects now contain multiple boards. Tasks will belong to a project, but will be displayed on specific boards.
*   `id` (PK)
*   `project_id` (FK -> projects.id)
*   `name` (String) - e.g., "Active Sprint", "Bug Triage".
*   `is_default` (Boolean) - Which board to load first.
*   `timestamps`

### 2.3 The `tasks` Table Update
Tasks must decouple from the global ID for display purposes and link to boards.
*   `id` (PK) - Keep for internal relational mapping.
*   `project_id` (FK -> projects.id) - Strict boundary.
*   `board_id` (FK -> boards.id, Nullable) - Which board it currently sits on. **[NEW]**
*   `project_task_number` (Integer) - The sequential number within the project. **[NEW]**
*   *(... other columns remain the same: title, status, assignee_id, etc.)*

## 3. Backend Logic & Application Layer

### 3.1 Task Creation Transaction (CRITICAL)
Generating the `CORE-142` style ID requires preventing race conditions when two users create tasks in the same project simultaneously.

**Logic (Eloquent `creating` Hook or Service Class):**
```php
DB::transaction(function () use ($project, $taskData) {
    // 1. Lock the project row to prevent race conditions on the counter
    $lockedProject = Project::where('id', $project->id)->lockForUpdate()->first();
    
    // 2. Increment the counter
    $lockedProject->increment('task_sequence');
    
    // 3. Create the task with the new sequence number
    $taskData['project_task_number'] = $lockedProject->task_sequence;
    return $lockedProject->tasks()->create($taskData);
});
```
*   **Virtual Attribute:** Add an accessor to the `Task` model to easily get the formatted ID:
    ```php
    public function getFormattedIdAttribute() {
        return "{$this->project->key}-{$this->project_task_number}";
    }
    ```

### 3.2 Routing & Controller Restructure
Routes must now reflect the deep hierarchy.
*   `GET /projects/{project_key}/boards/{board_id?}` - Primary view.
*   Inertia Controller must pass:
    *   `auth_user`
    *   `current_project` (including `key`)
    *   `boards` (list for the tabs)
    *   `columns` (with tasks joined to the active board)
