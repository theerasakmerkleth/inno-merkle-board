## 1. Product Vision & Strategy
TaskFlow AI is designed to be the definitive enterprise task management platform, bridging the gap between human creativity and AI efficiency. Adopting a **"Minimal Enterprise Grade"** design philosophy, it maximizes data density while providing a seamless "Human-in-the-loop" experience. 

**[NEW STRATEGIC PIVOT]** To truly serve enterprise scale, the platform must break away from a single global workspace. It must adopt a **Multi-Project Portfolio** structure, where a single enterprise can host dozens of isolated Projects, and each Project can contain multiple custom Boards (e.g., "Sprint 1 Board", "Backlog Board", "Bug Triage Board"), mirroring industry best practices found in Jira and Linear.

**Target Audience:** Enterprise Software Teams, DevOps Engineers, System Administrators, and Autonomous AI Workers.

**Pain Points Addressed:**
- Lack of granular access control and data security across multiple teams.
- **[NEW]** Inability to manage multiple independent products/teams within the same instance.
- **[NEW]** Cluttered workspaces due to having only one massive global Kanban board.

## 2. User Stories & Acceptance Criteria (AC)

### 2.1 Multi-Project Portfolio (Jira/Linear style)
**Story (US-008) [NEW]:** *As a PMO Director, I want the workspace to support multiple independent Projects, so that different teams (e.g., Marketing, Core Backend) can work in isolation.*
*   **AC 1:** A user can create a new `Project` with a unique Key/Prefix (e.g., `CORE`, `MKT`).
*   **AC 2:** Task IDs must be prefixed by the Project Key (e.g., `CORE-102` instead of just `102`).
*   **AC 3:** Users have a "Project Switcher" in the main navigation to jump between projects they have access to.

### 2.2 Multi-Board Architecture per Project
**Story (US-009) [NEW]:** *As a Project Manager, I want to create multiple Boards within a single Project, so that I can organize tasks into Sprints, Backlogs, or specific workflows.*
*   **AC 1:** A `Project` can have multiple `Boards` (e.g., "Sprint 22", "Active Bugs").
*   **AC 2:** Each `Board` defines its own set of `Columns` (Statuses) mapping to the underlying workflow.
*   **AC 3:** Tasks belong to a Project, but can be displayed on specific Boards based on filters (e.g., Sprint association). For MVP, a task belongs directly to a Board.

### 2.3 System Administrator - Global Security & Auth
*(Unchanged from v5.0 - US-001, US-002)*
*   Admin manages Global Roles (`Admin`, `Project Manager`, `Developer`, `QA`, `AI Agent`).
*   System utilizes standard Laravel session Auth.

### 2.4 Project Manager (PM) - Project-Level Control
*(Updated for Multi-Project)*
**Story (US-003):** *As a PM, I want to manage my project's team members, so I can ensure secure delivery.*
*   **AC 1:** PM assigns workspace users to specific `Projects` with Project-Level Roles (`Manager`, `Contributor`, `Viewer`).
*   **AC 2:** Users cannot access a Project's Boards unless they are added to the `project_user` list (or are Global Admins).

### 2.5 Developer / QA & Telemetry
*(Unchanged from v5.0 - US-004, US-005, US-007)*
*   Task status changes (drag-and-drop) are restricted by Role (`can_move_to_done`).
*   All changes are logged to `task_transitions` for Cycle Time calculation.
*   UI follows the Minimal Enterprise (Swiss Minimalist) design spec.

## 4. UI/UX Design Guidelines (Ref: Minimal Enterprise Grade)
- **Navigation:** Introduce a collapsible left-sidebar or top-left dropdown for the **Project Switcher**.
- **Board Navigation:** Inside a project, use minimal top-tabs (shadcn `<Tabs>`) to switch between different Boards (e.g., Backlog, Active Sprint).
- **Task Cards:** Prefix IDs prominently (`CORE-142` instead of `#142`) using `font-mono text-zinc-500`.
