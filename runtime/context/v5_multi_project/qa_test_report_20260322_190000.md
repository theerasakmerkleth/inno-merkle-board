## 1. PHASE 1: TEST STRATEGY & RISK ANALYSIS

The system underwent a major architectural pivot in Sprint 3 to support Jira-style Multi-Project Portfolio navigation. This introduced significant regression risks that required validation.

| Risk Category | Identified Risk | Test Approach / Mitigation |
| :--- | :--- | :--- |
| **Data Integrity** | **Task ID Race Conditions:** With the new `project_task_number` counter, simultaneous task creations might result in duplicate `CORE-1` IDs. | **Architectural Review:** Confirmed the Developer implemented `DB::transaction()` with `lockForUpdate()` in the `TaskObserver`. This forces database-level serialization on project rows, preventing race conditions. |
| **Routing & Auth** | **Dead Ends (404s):** The root route `/` was changed to redirect users to their assigned projects. Users with no projects or misconfigured roles might hit a dead end. | **Test Case:** Updated `test_authenticated_user_can_access_dashboard` to verify the redirect logic correctly resolves the active `project_key` based on the user's role. |
| **Security (RBAC)**| **Project Access Isolation:** A user assigned to project `MKT` should not be able to guess the URL and access `/projects/CORE/boards`. | **Code Review & Simulation:** Web routes include authorization logic verifying `$project->users->contains($user->id)`. Global Admins bypass this. |
