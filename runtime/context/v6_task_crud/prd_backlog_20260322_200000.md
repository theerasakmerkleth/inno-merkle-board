## 1. Product Vision & Strategy
TaskFlow AI is designed to be the definitive enterprise task management platform. Following the successful implementation of the **Multi-Project Portfolio** (v6.0), the platform now has a robust structural foundation. 

**[CURRENT STRATEGIC FOCUS]** The next critical phase is to flesh out the **Task Management Lifecycle**. Currently, tasks can only be moved via drag-and-drop. To be a viable Jira/Linear alternative, users must have full CRUD (Create, Read, Update, Delete) capabilities for tasks, complete with rich text descriptions, assignment, and rigorous permission checks.

**Target Audience:** Enterprise Software Teams, DevOps Engineers, System Administrators, and Autonomous AI Workers.

## 2. User Stories & Acceptance Criteria (AC)

### 2.1 Task Creation (Create)
**Story (US-010):** *As a Team Member, I want to create a new task within a specific project and board, so that I can add new work to the pipeline.*
*   **AC 1:** A "Create Task" button is available in the Board header. Clicking it opens a modal or slide-over panel.
*   **AC 2:** The creation form must include: Title (Required), Description (Rich Text/Markdown), Priority, and Assignee (Optional).
*   **AC 3:** Upon creation, the task is automatically assigned to the `todo` status column of the current board and generates a unique ID (e.g., `CORE-143`).
*   **AC 4:** *Role Check:* Only users with `Manager` or `Contributor` project roles can create tasks. `Viewer` roles cannot.

### 2.2 Task Details & Editing (Read/Update)
**Story (US-011):** *As a Team Member, I want to click on a task card to view its full details and edit them, so that I can update requirements or reassign the work.*
*   **AC 1:** Clicking a Task Card opens a detailed view (Modal or dedicated page).
*   **AC 2:** Users can edit the Title, Description, Priority, and Assignee.
*   **AC 3:** Users can manually change the Status via a dropdown (which must respect the same RBAC rules as drag-and-drop).
*   **AC 4:** *Role Check:* Only `Manager` or `Contributor` roles can edit. `Viewer` roles can only read.

### 2.3 Task Assignment (Assign)
**Story (US-012):** *As a PM or Developer, I want to assign a task to a specific human user or an AI Agent, so that responsibility is clearly defined.*
*   **AC 1:** The Assignee dropdown must only list users who are members of the current Project (plus Global Admins/AI Agents).
*   **AC 2:** If an AI Agent is assigned, the `is_ai_assigned` flag is set to true, triggering the AI's internal polling mechanism.

### 2.4 Task Deletion (Delete)
**Story (US-013):** *As a Project Manager, I want to delete obsolete or duplicate tasks, so that the board remains clean.*
*   **AC 1:** A "Delete" action is available within the Task Detail view.
*   **AC 2:** Deleting a task prompts a confirmation dialog to prevent accidental data loss.
*   **AC 3:** *Role Check:* **ONLY** users with the `Manager` project role (or Global `Admin`) can delete tasks. `Contributor` and `Viewer` roles cannot.

## 4. UI/UX Design Guidelines (Ref: Minimal Enterprise Grade)
- **Modals/Slide-overs:** Use `shadcn/Sheet` (slide-over from the right) or `shadcn/Dialog` for Task Creation and Editing to keep the user in the context of the Board.
- **Rich Text:** Keep the description editor minimal. Support basic Markdown.
- **Forms:** Adhere to the "Minimal Enterprise" spec—no heavy borders, use subtle inputs, and rely on whitespace.

## 5. Handover to Developer & Solution Architect
**Feature Requirements Checklist for Developer:**

1.  **Backend (Laravel):**
    *   **Create:** `POST /projects/{project}/boards/{board}/tasks` (Validate input, handle `task_sequence` increment safely).
    *   **Read/Update:** `PATCH /tasks/{id}` (Update details, re-assign).
    *   **Delete:** `DELETE /tasks/{id}` (Enforce `Manager` role policy).
    *   **Assignment Data:** The Inertia payload for the Kanban board must now include an array of `project_members` so the frontend can render the Assignee dropdown.
    *   **Policies/FormRequests:** Implement strict Authorization checks (e.g., `TaskPolicy`) to enforce the ACs (Viewers can't edit, Contributors can't delete).

2.  **Frontend (React/Inertia/shadcn):**
    *   **Create Task Modal:** Add a button in the Board Header that opens a form (Title, Desc, Priority, Assignee).
    *   **Task Detail Modal:** Make Task Cards clickable. Open a detailed view allowing inline editing of the task fields.
    *   **Delete Button:** Add a red "Delete Task" button inside the Task Detail view, visible only to authorized roles.
    *   **State Management:** Ensure Inertia form submissions smoothly update the UI without full page reloads (`preserveScroll`, `preserveState`).
