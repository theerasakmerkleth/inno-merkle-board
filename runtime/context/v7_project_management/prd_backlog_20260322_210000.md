## 1. Product Vision & Strategy
TaskFlow AI is designed to be the definitive enterprise task management platform. While we have successfully implemented the core multi-project structural changes (v6.0) and Task CRUD (v7.0), the platform lacks the administrative tools necessary for a Project Manager (PM) or Global Admin to independently spin up new projects or manage team rosters.

**[CURRENT STRATEGIC FOCUS]** The objective for this sprint is to empower PMs and Admins with full **Project Portfolio Management (CRUD)** and **Project Member Management** capabilities. This removes the reliance on backend seeding and allows the enterprise to scale dynamically.

**Target Audience:** Project Managers, System Administrators.

## 2. User Stories & Acceptance Criteria (AC)

### 2.1 Project Creation & Configuration (Create/Update)
**Story (US-014):** *As a Global Admin or PM, I want to create a new Project and configure its details, so that I can onboard a new product team.*
*   **AC 1:** A "New Project" button is accessible from the global navigation or project switcher.
*   **AC 2:** Creating a project requires a `Name` and a unique `Project Key` (e.g., 3-5 uppercase letters like `MKT`).
*   **AC 3:** Editing a project (via a "Project Settings" page) allows changing the Name and Status (Active/Archived). 
*   **AC 4:** *Role Check:* Only Global Admins or users with the `create_projects` global permission can create new projects. Only Project Managers can edit their respective projects.

### 2.2 Project Deletion/Archiving (Delete)
**Story (US-015):** *As a Global Admin or PM, I want to archive or delete a project, so that the workspace remains uncluttered.*
*   **AC 1:** Deleting a project permanently removes all associated Boards, Tasks, and Memberships. It requires a strict confirmation dialog (e.g., typing the project name).
*   **AC 2:** Archiving a project hides it from default views but preserves the data.
*   **AC 3:** *Role Check:* Only Global Admins or the Project's `Manager` can archive/delete the project.

### 2.3 Project Member Management (RBAC Assignment)
**Story (US-016):** *As a Project Manager, I want to add or remove team members from my project and assign them roles, so that I can control access and responsibilities securely.*
*   **AC 1:** A "Team Settings" or "Members" page exists within the project context.
*   **AC 2:** PMs can search for existing Workspace Users and add them to the project.
*   **AC 3:** Upon adding, or at any time after, the PM can set the user's project role (`Manager`, `Contributor`, `Viewer`).
*   **AC 4:** PMs can remove a user from the project, which strips their access to the project's boards entirely.

### 2.4 Multi-Board Management (Bonus/Included)
**Story (US-017):** *As a Project Manager, I want to create and delete boards within my project, so that I can manage different workflows (e.g., Sprints vs. Backlog).*
*   **AC 1:** A button to create a new Board within the current project.
*   **AC 2:** Ability to delete a board (which moves tasks back to an unassigned board state or deletes them based on implementation choice).

## 4. UI/UX Design Guidelines (Ref: Minimal Enterprise Grade)
- **Project Settings Page:** Introduce a dedicated `/projects/{key}/settings` view with minimal, cleanly separated sections (General, Members, Danger Zone).
- **Member Management:** Use an edge-to-edge list or table similar to the Global User Management (`/users`), but specifically scoped to the project. Include a minimalist combobox or slide-over to search and add new users.
- **Danger Zone:** Follow standard SaaS conventions: red borders and explicit warning texts for destructive actions like Project Deletion.

## 5. Handover to Developer & Solution Architect

**Feature Requirements & Function Checklist for Developer:**

**Backend (Laravel APIs & Logic):**
1.  **ProjectController:**
    *   `POST /projects`: Create project. Auto-create a default "Main Board". Assign the creator as the `Manager` in the `project_user` table.
    *   `PATCH /projects/{project}`: Update project details.
    *   `DELETE /projects/{project}`: Delete project.
2.  **ProjectMemberController:**
    *   `GET /projects/{project}/members`: Return list of members.
    *   `POST /projects/{project}/members`: Attach a user with a specific role.
    *   `PATCH /projects/{project}/members/{user}`: Update a user's role.
    *   `DELETE /projects/{project}/members/{user}`: Detach a user.
3.  **BoardController:**
    *   `POST /projects/{project}/boards`: Create a new board.
    *   `DELETE /projects/{project}/boards/{board}`: Delete a board.
4.  **Authorization (Policies):**
    *   Create `ProjectPolicy` and `BoardPolicy` to strictly enforce that only `Manager` roles (or Global Admins) can perform CRUD on Project details, Roster, and Boards.

**Frontend (React/Inertia):**
1.  **Global UI:** Add a "Create Project" button in the Left Sidebar (visible based on global permissions).
2.  **Settings Layout:** Create a new page `Pages/Projects/Settings.tsx` to house the configuration.
3.  **Members Tab:** A sub-section in Settings displaying the current roster, with an interface to add workspace users.
4.  **Board Management:** Add a generic "+ Create Board" tab in the `KanbanBoard.tsx` header area.
