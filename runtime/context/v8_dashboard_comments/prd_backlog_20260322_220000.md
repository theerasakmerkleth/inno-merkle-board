## 1. Product Vision & Strategy
TaskFlow AI has established a robust structural foundation with Multi-Project Portfolios (v6.0), Task CRUD capabilities (v7.0), and dynamic Project/Team Management (v8.0). 

**[CURRENT STRATEGIC FOCUS]** The platform is structurally sound but functionally siloed. To be a viable daily-use Jira/Linear alternative for enterprises, we must bridge the communication gap and provide portfolio-level visibility. 
1. **Collaboration:** Users and AI Agents need a place to discuss tasks and post updates (Comments).
2. **Visibility:** PMOs and users need an aggregated view of their work and team performance across all isolated projects (Enterprise Dashboard / My Tasks).

**Target Audience:** Enterprise Software Teams, Project Managers, PMO Analysts, and Autonomous AI Workers.

## 2. User Stories & Acceptance Criteria (AC)

### 2.1 Task Comments & Activity (Collaboration)
**Story (US-018):** *As a Team Member or AI Agent, I want to add comments to a task, so that I can discuss requirements, post updates, or log automated test results.*
*   **AC 1:** A "Comments" section exists within the Task Detail modal/view.
*   **AC 2:** Users can post rich-text or plain-text comments.
*   **AC 3:** Comments display the author's name, avatar, and a relative timestamp (e.g., "2 hours ago").
*   **AC 4:** AI Agents can post comments via the API. These comments should be visually distinct (e.g., a subtle highlight or AI icon) to differentiate them from human input.
*   **AC 5:** *Role Check:* Only `Manager` and `Contributor` project roles can post comments. `Viewer` roles can read comments but cannot post.

### 2.2 Global "My Tasks" Inbox (Visibility)
**Story (US-019):** *As a Team Member, I want a single unified view of all tasks assigned to me across every project, so that I can prioritize my daily work without digging through multiple boards.*
*   **AC 1:** A new global navigation item "My Tasks" or "Inbox" exists in the main sidebar.
*   **AC 2:** The view displays a list or lightweight Kanban board containing *only* tasks where `assignee_id` matches the current user.
*   **AC 3:** Tasks in this view prominently display their originating Project Key (e.g., `[CORE]`).
*   **AC 4:** Clicking a task in this view opens the standard Task Detail modal, allowing full CRUD interactions based on the user's project-specific role.

### 2.3 Enterprise Resource Dashboard (Analytics)
**Story (US-020):** *As a PMO Director or Manager, I want a high-level dashboard showing resource allocation, so that I can identify bottlenecks and reassign overloaded team members.*
*   **AC 1:** A "Dashboard" or "Analytics" page exists at the global level.
*   **AC 2:** The dashboard displays a "Resource Loading" summary (e.g., a table or simple bar chart showing how many 'In Progress' or 'To Do' tasks are assigned to each user across the workspace).
*   **AC 3:** The dashboard calculates and displays the average "Cycle Time" (time from `in_progress` to `done`) by querying the `task_transitions` telemetry table.
*   **AC 4:** *Role Check:* Accessible to Global `Admin`, `Project Manager`, or users with a global `view_analytics` permission.

## 4. UI/UX Design Guidelines (Ref: Minimal Enterprise Grade)
- **Comments UI:** Keep it clean. Use an edge-to-edge list within the Task Modal. Do not use heavy chat bubbles; use a simple vertical timeline layout with borders separating messages.
- **My Tasks View:** This should be the default landing page (`/`) when a user logs in, replacing the current behavior of redirecting to a random project. It should look like a sleek, high-density data table or list.
- **Dashboard:** Use standard `shadcn/ui` Cards to frame key metrics. Avoid overly complex charts for the MVP; simple numeric aggregations and basic progress bars (`shadcn/Progress`) are sufficient.

## 5. Handover to Service Designer & Solution Architect

**Feature Requirements & Function Checklist for Developer:**

**Backend (Laravel APIs & Logic):**
1.  **Comments Schema & API:**
    *   Create `Comment` model and migration (`task_id`, `user_id`, `content`).
    *   API: `POST /tasks/{task}/comments` and `GET /tasks/{task}/comments`.
    *   Add `CommentPolicy` (Contributors/Managers can create).
2.  **My Tasks API:**
    *   Update `routes/web.php` so the root `/` route loads the "My Tasks" view instead of a project redirect.
    *   Query: `Task::where('assignee_id', auth()->id())->with('project')->get()`.
3.  **Dashboard API:**
    *   Create `DashboardController` handling aggregations.
    *   Query 1 (Resource Load): Count tasks grouped by `assignee_id` where status is not `done`.
    *   Query 2 (Cycle Time): Calculate average time difference between `in_progress` and `done` records in `task_transitions`.

**Frontend (React/Inertia):**
1.  **Task Modal Update:** Add a Comments section below the description in `TaskModal`. Fetch comments and render the timeline.
2.  **My Tasks Page:** Build `Pages/Dashboard/MyTasks.tsx`.
3.  **Analytics Page:** Build `Pages/Dashboard/Analytics.tsx` to visualize the PMO metrics.
